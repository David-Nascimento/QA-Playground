6. Casos de Teste Práticos

Casos de teste reais e acionáveis para validar acessibilidade em diferentes cenários de uso.

Casos de teste reais e acionáveis para validar acessibilidade em diferentes cenários de uso.

Caso de Teste 1: Navegação por Teclado em Formulário

Cenário: Usuário navega no formulário de contato apenas com teclado.

Passos:

  1. Usuário acessa a página do formulário
  2. Pressiona Tab para navegar pelos campos
  3. Preenche cada campo usando apenas o teclado
  4. Navega até o botão "Enviar"
  5. Pressiona Enter para submeter

Resultado Esperado:

  • Foco visível em todos os campos
  • Ordem de tabulação segue: Nome → Email → Mensagem → Enviar
  • Labels são anunciados ao focar cada campo
  • Mensagens de erro aparecem e são associadas aos campos
  • Formulário pode ser submetido completamente via teclado

Critério WCAG Relacionado:

  • 2.1.1 Teclado: Todo funcionalidade é acessível via teclado (Nível A)
  • 2.4.3 Ordem de Foco: Ordem de foco preserva significado (Nível A)
  • 4.1.2 Nome, Função, Valor: Nomes acessíveis para componentes (Nível A)

Caso de Teste 2: Leitor de Tela em Formulário Obrigatório

Cenário: Usuário com leitor de tela preenche um campo obrigatório e recebe feedback de erro.

Passos:

  1. Usuário inicia leitor de tela (NVDA/JAWS/VoiceOver)
  2. Navega até o campo "Email" (obrigatório)
  3. Deixa o campo vazio e tenta submeter
  4. Leitor de tela anuncia mensagem de erro
  5. Navega de volta ao campo para corrigir

Resultado Esperado:

  • Campo obrigatório é identificado (aria-required="true" ou indicação visual)
  • Mensagem de erro é anunciada ao tentar submeter
  • Erro é associado ao campo via aria-describedby
  • Foco retorna ao campo com erro para correção
  • Mensagem de erro desaparece quando campo é corrigido

Critério WCAG Relacionado:

  • 3.3.1 Identificação de Erro: Erros são identificados e descritos (Nível A)
  • 3.3.2 Rótulos ou Instruções: Rótulos ou instruções são fornecidos (Nível A)
  • 4.1.3 Mensagens de Status: Status é programaticamente determinado (Nível AA)

Caso de Teste 3: Zoom 200% em Página Mobile

Cenário: Usuário com baixa visão aumenta zoom para 200% em dispositivo móvel.

Passos:

  1. Usuário acessa a página no dispositivo móvel
  2. Aumenta zoom para 200% (Configurações → Tela → Tamanho da fonte ou gestos de pinça)
  3. Tenta navegar pela página
  4. Tenta preencher formulários
  5. Tenta clicar em botões e links

Resultado Esperado:

  • Todo conteúdo permanece legível sem rolagem horizontal
  • Layout não quebra ou sobrepõe elementos
  • Elementos interativos têm tamanho mínimo de 44x44px
  • Navegação continua funcional
  • Texto pode ser redimensionado sem perda de funcionalidade

Critério WCAG Relacionado:

  • 1.4.4 Redimensionamento de Texto: Texto pode ser redimensionado até 200% (Nível AA)
  • 1.4.10 Refluxo: Conteúdo pode ser apresentado sem perda de informação (Nível AA)
  • 2.5.5 Tamanho de Alvo: Tamanho mínimo de alvo para ponteiro (Nível AAA)

Caso de Teste 4: Imagens Sem Texto Alternativo

Cenário: Usuário com leitor de tela navega por página com imagens informativas.

Passos:

  1. Usuário inicia leitor de tela
  2. Navega pela página encontrando várias imagens
  3. Leitor de tela anuncia cada imagem
  4. Verifica se texto alternativo está presente

Resultado Esperado:

  • Imagens informativas têm alt descritivo
  • Imagens decorativas têm alt="" ou aria-hidden="true"
  • Textos alternativos são concisos e descritivos
  • Imagens funcionais (links/botões) têm alt que descreve a ação

Critério WCAG Relacionado:

  • 1.1.1 Conteúdo Não-textual: Alternativas textuais para conteúdo não-textual (Nível A)

Caso de Teste 5: Modal com Foco Gerenciado

Cenário: Usuário abre modal e navega apenas com teclado, depois fecha.

Passos:

  1. Usuário pressiona Tab até o botão "Abrir Modal"
  2. Pressiona Enter para abrir modal
  3. Verifica se foco vai para dentro do modal
  4. Navega pelo conteúdo do modal com Tab
  5. Tenta pressionar Tab para sair do modal (deve ficar preso)
  6. Pressiona Esc para fechar modal
  7. Verifica se foco retorna ao botão que abriu o modal

Resultado Esperado:

  • Foco move para dentro do modal ao abrir
  • Foco fica "preso" no modal (não escapa para conteúdo de fundo)
  • Esc fecha o modal
  • Foco retorna ao elemento que abriu o modal
  • Conteúdo de fundo é inacessível (aria-hidden) quando modal está aberto

Critério WCAG Relacionado:

  • 2.1.2 Sem Armadilha de Teclado: Foco não fica preso (Nível A)
  • 2.4.3 Ordem de Foco: Ordem de foco preserva significado (Nível A)
  • 4.1.3 Mensagens de Status: Status é programaticamente determinado (Nível AA)
